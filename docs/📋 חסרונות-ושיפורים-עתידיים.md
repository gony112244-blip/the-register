## 🔴 עדיפות גבוהה — לטפל בתוך חודשיים

### 1. JWT Secret חלש מדי

**הבעיה:**  
ה-JWT Secret הנוכחי הוא `superSecretKey123` — סיסמה פשוטה וצפויה. אם תקיף מנחש אותה, הוא יכול לזייף tokens ולהתחזות לכל משתמש, כולל אדמין.

**הסיכון:** גישה בלתי מורשית לחשבונות.

**הפתרון:**
```bash
# ייצור secret חזק:
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
# להכניס את התוצאה ב-.env: JWT_SECRET=<הערך החדש>
# ואז: pm2 restart hapinkas --update-env
```
**שים לב:** שינוי ה-Secret מנתק את כל המשתמשים המחוברים — יצטרכו להתחבר מחדש.

---

### 2. אין Pagination (הגבלת תוצאות) על רשימות

**הבעיה:**  
המסלולים `/my-requests` ו-`/my-sent-requests` מחזירים את **כל הרשומות** בלי הגבלה. עם 50 בקשות — לא בעיה. עם 500 בקשות — השרת שולח ספר שלם לכל בקשה.

**הסיכון:** איטיות גדלה עם הזמן, בזבוז זיכרון.

**הפתרון:**  
להוסיף `LIMIT` + `OFFSET` לשאילתות ו-pagination ב-frontend.
```sql
-- דוגמה:
SELECT ... FROM connections WHERE ... ORDER BY created_at DESC LIMIT 20 OFFSET $2
```

### 3. Race Condition קטן ב-`/finalize-connection`

**הבעיה:**  
כששני צדדים מאשרים את תנאי התשלום בדיוק באותה מילי-שנייה, ייתכן שכל אחד מקבל הודעת "ממתינים לצד השני" במקום "שניכם אישרתם". הסטטוס בסוף יתעדכן נכון ב-DB, אבל ההודעה למשתמש תהיה מבלבלת.

**הסיכון:** נמוך — מצריך תזמון מושלם של שני אנשים.

**הפתרון:**  
לעטוף את פעולת הfinalize בטרנזקציה עם `SELECT FOR UPDATE`:
```javascript
const client = await pool.connect();
await client.query('BEGIN');
// ... לוגיקה ...
await client.query('COMMIT');
```

## 🟡 עדיפות בינונית — לפני שמגיעים ל-500 משתמש

### 4. ביצועי שאילתת ההתאמות — `NOT IN` לעומת `NOT EXISTS`

**הבעיה:**  
מנוע ההתאמות (`match-engine.js`) משתמש ב-`NOT IN (SELECT ...)` למסנן חיבורים, חסימות והסתרות. ב-PostgreSQL, `NOT IN` על טבלאות גדולות (1,000+ שורות) הרבה יותר איטי מ-`NOT EXISTS`.

**הסיכון:** ביצועים יידרדרו בהדרגה ככל שהטבלאות יגדלו.

**הפתרון:**  
להמיר את כל ה-`NOT IN` ל-`NOT EXISTS` ב-`match-engine.js`:
```sql
-- במקום:
WHERE id NOT IN (SELECT receiver_id FROM connections WHERE sender_id = $1)

-- כך:
WHERE NOT EXISTS (SELECT 1 FROM connections WHERE sender_id = $1 AND receiver_id = users.id)
```
**מורכבות:** בינונית — שבוע עבודה עם בדיקות.

---

### 5. אין Cache לתוצאות ההתאמות

**הבעיה:**  
כל פעם שמשתמש נכנס לדף ה-Matches, השרת מריץ שאילתה כבדה מ-DB מאפס. אם המשתמש מרענן את הדף 10 פעמים — 10 שאילתות כבדות.

**הסיכון:** עומס מיותר על ה-DB בשעות שיא.

**הפתרון:**  
שמירת תוצאות ב-in-memory cache ל-5 דקות, עם איפוס כשמשתמש שולח/מקבל פנייה.

---

### 6. TTS Index כקובץ JSON — בעייתי בקלסטר

**הבעיה:**  
הקובץ `ivr-audio/tts-index.json` נכתב ע"י כל Worker ב-PM2 בנפרד עם `fs.writeFileSync`. אם שני Workers מייצרים TTS חדש בו-זמנית, האחרון שכותב "מנצח" ורשומת ה-Worker הראשון עלולה לאבד.

**הסיכון:** נמוך בפועל — מתרחש רק ביצירת TTS חדש (לא מהcache). הפגיעה: אותו משפט יוצר מחדש פעם נוספת.

**הפתרון:**  
לאחסן את ה-TTS index ב-DB במקום בקובץ JSON:
```sql
CREATE TABLE tts_cache (
    hash VARCHAR(12) PRIMARY KEY,
    yemot_path TEXT,
    text_preview TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);
```

## 🟢 עדיפות נמוכה — "שיהיה יפה" — לאחר שהכל יציב

### 7. מעבר לשרת גדול יותר

**הבעיה:**  
השרת הנוכחי הוא VPS עם **ליבה אחת** ו-~11% שימוש בזיכרון כרגע. עם גדילה, ליבה אחת תהפוך לצוואר בקבוק.

**הסימן לשדרוג:**  
כאשר `pm2 monit` מציג CPU > 70% באופן קבוע, או Memory > 400MB.

**הפתרון:**  
שדרוג VPS ל-2–4 ליבות ב-Hostinger (עלות: כ-₪50-100/חודש). PM2 יתחיל להשתמש בכל הליבות אוטומטית.

---

### 8. Helmet HSTS מושבת

**הבעיה:**  
ב-`server.js`, ה-`helmet` מוגדר עם `hsts: false`. HSTS היא הגדרה שאומרת לדפדפן "תמיד השתמש ב-HTTPS עם אתר זה".

**למה זה כך:** כנראה הוגדר כשה-HTTPS עבר דרך Nginx ולא ישירות ב-Node.

**הפתרון:**  
אם Nginx מטפל ב-SSL (סביר), ניתן להגדיר HSTS ב-Nginx במקום:
```nginx
add_header Strict-Transport-Security "max-age=31536000" always;
```

---

### 9. אין Fallback אם ימות המשיח נופל

**הבעיה:**  
המערכת הטלפונית תלויה לחלוטין בשרתי ימות המשיח. אם השירות שלהם יורד (נדיר, אבל אפשרי) — ה-IVR לא עובד.

**הסיכון:** נמוך — ימות הם שירות אמין. אבל אין Plan B.

**הפתרון:**  
אין פתרון טכני פשוט — ימות הוא ה-backbone. אפשרות: להוסיף הודעה ב-webhook timeout שתשלח SMS/מייל לאדמין.

---

### 10. Session IVR — שתי שיחות בו-זמנית מאותו משתמש

**הבעיה:**  
`getOrCreateSession` מגדיר `ON CONFLICT (user_id) DO UPDATE` — כלומר session אחד למשתמש. אם אותו אדם מתקשר משני מכשירים שונים בו-זמנית (נדיר מאוד), השיחה השנייה תדרוס את מצב השיחה הראשונה.

**הסיכון:** נמוך מאוד — תרחיש כמעט תיאורטי.

**הפתרון:**  
לשנות את ה-UNIQUE key מ-`user_id` ל-`enter_id` (מזהה שיחה ייחודי שימות שולחים).

---

### 11. לוגים — אין התראה על שגיאות קריטיות

**הבעיה:**  
כל `catch` ב-server.js מדפיס לקונסול אבל לא שולח התראה. אם יש גל של שגיאות 500 בלילה — תדע רק בבוקר כשתבדוק את הלוגים.

**הפתרון:**  
ניתן להוסיף פונקציה קטנה:
```javascript
function alertAdmin(msg) {
    if (process.env.NODE_ENV === 'production') {
        // שלח מייל / SMS לאדמין
        sendEmail(ADMIN_EMAIL, 'שגיאה קריטית', msg);
    }
}
```
ולקרוא לה בשגיאות קריטיות בלבד (DB down, IVR crash).
