# אבחון ותכנון: גיבוי מסד נתונים וביצועי אתר

**תאריך:** אפריל 2026  
**מטרה:** הצגת מצב קיים, סיכוני, והמלצות פרקטיות לשתי נקודות תשתיתיות.

---

## חלק א' — גיבוי מסד הנתונים

### מצב קיים

| פרט | מה יש |
|-----|--------|
| **כלי גיבוי** | `backup_database.js` — מריץ `pg_dump` לקובץ SQL |
| **שמירה** | תיקיית `backups/` **על אותו שרת** שבו האתר |
| **אוטומציה** | ידנית בלבד (הרצת `npm run db:backup`) |
| **קובץ קיים** | גיבוי אחד מ-23.3.2026 — כ-120KB |
| **שמירה לאורך זמן** | עד 30 קבצים, אחר כך נמחקים |

### מה הסיכון הקיים

**הבעיה המרכזית:** הגיבוי נשמר **באותו מחשב שבו האתר רץ**.

אם קורה משהו לשרת — אסון, פריצה, תקלת דיסק, מחיקה בטעות — **גם הגיבוי נעלם עם הנתונים**. זה כאילו שאין גיבוי בכלל.

**בעיות נוספות:**
- גיבוי **ידני** = כנראה לא יקרה באופן קבוע
- גיבוי אחד מלפני חודש = חודש של נתונים שיאבדו אם קורה משהו היום
- אין גיבוי **מחוץ לשרת** בכלל

### המלצות — לפי עלות ופשטות

#### אפשרות 1 — Cron + Google Drive (חינם, פשוט ביותר)

**עלות:** 0 ₪  
**מורכבות:** נמוכה — כשעה הגדרה  

**איך עובד:**
1. Cron Job בשרת (לינוקס) מריץ את `backup_database.js` כל לילה בשעה קבועה
2. סקריפט קטן מעלה את הקובץ ל-**Google Drive** (15GB חינם)

```
# דוגמה ל-cron — כל לילה ב-02:00
0 2 * * * cd /var/www/hapinkas && npm run db:backup && node upload-to-gdrive.js
```

**יתרון:** קובץ מחוץ לשרת, חינמי, אוטומטי  
**חיסרון:** דורש הגדרת Google API חד פעמית

---

#### אפשרות 2 — Cron + Backblaze B2 (כמעט חינם)

**עלות:** ~$0.006 לGB לחודש — **פחות מ-1 אגורה לGB**  
**מורכבות:** נמוכה  

**איך עובד:** אותו עיקרון — Cron + העלאה ל-Backblaze B2 (שירות ענן אחסון).  
עבור מסד נתונים בגודל 120KB לגיבוי, 30 גיבויים = **פחות מ-0.5 ₪ לחודש**.

**יתרון:** זול, מהיר, ממשק נוח, CLI פשוט (`b2` command)  
**חיסרון:** דורש יצירת חשבון ב-Backblaze

---

#### אפשרות 3 — פתרון PostgreSQL מובנה: `pg_dump` + rclone (מומלץ ביותר)

**עלות:** 0–5 ₪/חודש  
**מורכבות:** בינונית — 2–3 שעות הגדרה  

**איך עובד:**
- `rclone` הוא כלי חינמי שמסנכרן תיקיות מקומיות לכל שירות ענן (Drive, Backblaze, S3, Dropbox)
- מגדירים פעם אחת: `rclone config` → בוחרים יעד
- ה-Cron מריץ גיבוי + sync אוטומטי

```bash
# Cron יומי
0 2 * * * npm run db:backup && rclone copy /var/www/hapinkas/backups remote:hapinkas-backups
```

**יתרון:** גמיש, אמין, ניתן לשנות יעד בלי לגעת בקוד  
**חיסרון:** הגדרה ראשונית טיפה יותר מורכבת

## חלק ב' — ביצועי האתר (איטיות)

### מצב קיים — מה גרם/גורם לאיטיות

לאחר בדיקת הקוד, זיהיתי שלוש סיבות עיקריות:

#### סיבה 1 — מנוע ההתאמות (`/matches`) — שאילתה כבדה

**מה קורה:** בכל פעם שמשתמש נכנס לדף ה-Matches, השרת:
1. שולף את כל פרטי המשתמש (`SELECT * FROM users`)
2. בונה שאילתה עם **10+ תנאי WHERE** דינמיים
3. עושה **6 תתי-שאילתות** (`NOT IN SELECT`) — connections, blocks, hidden, וכו'
4. כל אחת מהן סורקת טבלאות ללא אינדקסים

בינתיים ה-Pool של חיבורי ה-DB מוגדר **ללא מגבלות** (`db.js` — אין max connections), מה שאומר שעם עומס — החיבורים יצטברו.

#### סיבה 2 — אין אינדקסים על הטבלאות המרכזיות

בקוד קיים אינדקס **אחד בלבד**: על `activity_log`. טבלאות כמו `connections`, `photo_approvals`, `hidden_profiles`, `user_blocks` — **אין עליהן אינדקסים**.

כשמחפשים `WHERE sender_id = X` ב-`connections` בלי אינדקס — PostgreSQL עושה סריקה מלאה של כל הטבלה.

#### סיבה 3 — אין Cache ברמת ה-API

כל קריאה ל-`/matches` — גם אם המשתמש לא עשה כלום — יוצרת שאילתה מחדש. אין שמירה זמנית של תוצאות.

---

### האם זה יחמיר?

**כן — בוודאות**, כאשר:

| מצב | מה קורה |
|-----|---------|
| 50+ משתמשים פעילים בו-זמנית | שאילתות `/matches` מצטברות, השרת נחנק |
| טבלת `connections` גדלה ל-5,000+ שורות | ה-`NOT IN` בלי אינדקס הופך לסריקה ארוכה |
| מנהל נכנס לדשבורד + משתמשים רגילים | שאילתות כבדות מקבילות — DB Pool מתמלא |

**עכשיו** — כשיש מעט משתמשים — זה "עובד" אבל איטי.  
**בהמשך** — זה עלול לגרום לתגובות של 5–15 שניות, ואפילו קריסות.

---

### המלצות — לפי סדר עדיפות ומורכבות

#### פתרון 1 — אינדקסים (עלות: 0, מורכבות: נמוכה) — **הכי משתלם**

פעולה אחת שמשפרת ביצועים ב-50–80% בלי לגעת בלוגיקה.

להוסיף לפונקציית `updateDbSchema()` בשרת (בהמשך לאינדקס הקיים):

```sql
CREATE INDEX IF NOT EXISTS idx_connections_sender ON connections(sender_id);
CREATE INDEX IF NOT EXISTS idx_connections_receiver ON connections(receiver_id);
CREATE INDEX IF NOT EXISTS idx_connections_status ON connections(status);
CREATE INDEX IF NOT EXISTS idx_hidden_profiles_user ON hidden_profiles(user_id);
CREATE INDEX IF NOT EXISTS idx_user_blocks_blocker ON user_blocks(blocker_id);
CREATE INDEX IF NOT EXISTS idx_user_blocks_blocked ON user_blocks(blocked_id);
CREATE INDEX IF NOT EXISTS idx_photo_approvals_requester ON photo_approvals(requester_id);
CREATE INDEX IF NOT EXISTS idx_photo_approvals_target ON photo_approvals(target_id);
CREATE INDEX IF NOT EXISTS idx_users_gender_approved ON users(gender, is_approved, is_blocked);
```

**מורכבות יישום:** נמוכה — הוספת 9 שורות לקוד קיים.  
**תוצאה:** השאילתה הכבדה ב-`/matches` תרוץ פי 5–10 מהר יותר.

---

#### פתרון 2 — הגבלת Pool וחיבורי DB (עלות: 0, מורכבות: נמוכה)

כרגע ה-`db.js` לא מגביל כלום. להוסיף:

```javascript
const pool = new Pool({
  max: 20,                    // מקסימום 20 חיבורים במקביל
  idleTimeoutMillis: 30000,   // ניתוק חיבורים בטלים אחרי 30 שניות
  connectionTimeoutMillis: 5000, // timeout אם DB לא מגיב ב-5 שניות
});
```

**מורכבות יישום:** נמוכה — 3 שורות.  
**תוצאה:** מניעת "נחנקות" של השרת תחת עומס.

---

#### פתרון 3 — Cache פשוט לתוצאות `/matches` (עלות: 0, מורכבות: בינונית)

שמירת תוצאות מנוע ההתאמות בזיכרון (in-memory) לכ-5 דקות:

- אם משתמש קורא `/matches` 3 פעמים ברצף — שאילתת DB רצה **פעם אחת** בלבד
- Cache מתאפס כשמשתמש שולח/מקבל פנייה

**מורכבות יישום:** בינונית — שבוע עבודה קטן  
**תוצאה:** חיסכון של 70%+ בשאילתות DB בשעות שיא

---

#### פתרון 4 — שאילתת `/matches` — שינוי `NOT IN` ל-`NOT EXISTS` (מורכבות: בינונית)

`NOT IN` היא שיטה איטית ב-PostgreSQL עבור טבלאות גדולות. המרתה ל-`NOT EXISTS` משפרת ביצועים ב-30–50% על טבלאות עם 1,000+ שורות.  
**מורכבות:** שינוי פנימי בשאילתה — שבוע עבודה עם בדיקות.

---

### תכנית פעולה מומלצת

| שלב | מה לעשות | מתי | מורכבות |
|-----|----------|-----|---------|
| **1** | הוסף אינדקסים ל-`updateDbSchema()` | **עכשיו** | נמוכה |
| **2** | הוסף הגבלות Pool ב-`db.js` | **עכשיו** | נמוכה |
| **3** | Cache ל-`/matches` | לאחר ה-MVP | בינונית |
| **4** | שינוי `NOT IN` → `NOT EXISTS` | אחרי שיש 500+ משתמשים | בינונית |
| **5** | מעבר לשרת גדול יותר | אחרי 200+ משתמשים פעילים | נמוכה (עלות כסף) |

**שלבים 1+2 ביחד:** פחות מ-30 דקות עבודה, ואפס עלות — ועל כן **כדאי מאוד לעשות עכשיו** גם אם האתר "עובד".
