## 1. מטרה ועקרונות

### מה המערכת עושה
מערכת IVR המחוברת בזמן אמת לאתר "הפנקס", מאפשרת למשתמשי טלפון כשר לנהל את כל פעולות השידוך הפעילות שלהם — ללא גישה לאינטרנט.

### עקרונות מנחים

| עיקרון | פירוט |
|--------|--------|
| **Inbound בלבד** | המערכת מקבלת שיחות נכנסות בלבד. אין שיחות יוצאות, אין צינתוקים. |
| **ללא SMS** | מכשירים כשרים חסומים להודעות טקסט. ההתראות יתבססו על מיילים קיימים הכוללים משפט הפניה לחיוג. |
| **ממשק בלבד — לא לוגיקה** | ה-IVR שולח Webhooks לאותם API endpoints שמשמשים את האתר. אין שכפול של לוגיקה עסקית. |
| **קלילות וזולות** | מינימום הקראה, מינימום תפריטים מיותרים, מקסימום פעולה ישירה. |

## 2. ארכיטקטורה ושירותי צד שלישי

### ספק תקשורת: "ימות משיח"
- נבחר בשל התאמתו למגזר החרדי וההיכרות עם קווים כשרים.
- החיבור: API + Webhooks דו-כיווניים.
- ימות משיח שולח לשרת "הפנקס" את נתוני ההקשה → השרת מחזיר פקודות (השמע קובץ / מעבר שלב / ניתוק).

### מנוע TTS: Google Cloud TTS

> ⚠️ **החלטה מוצהרת — לא לשנות:**  
> **לא** משתמשים ב-TTS של ימות משיח — יקר ונמוך איכות.  
> **הזרימה הנכונה:** השרת שלנו מייצר MP3 דרך Google → שומר ב-Cache → שולח **URL** ב-JSON → ימות משיח רק מנגנים.

- שפה ומודל: `he-IL`, קולות Neural — הקראה זורמת ואנושית.
- מודל כלכלי: Free Tier (עד ~מיליון תווים/חודש) + מנגנון Cache מקומי שמוריד את הצריכה לאפס עם הזמן.

## 3. אבטחה וזיהוי

### זרימת הכניסה

```
חיוג נכנס
    │
    ▼
האם המספר מזוהה ב-DB?
    ├── לא → "הקש את מספר הנייד שלך" → "הקש PIN"
    └── כן → האם allow_ivr_no_pass = true?
                ├── כן → כניסה ישירה
                └── לא → "הקש PIN"
                              │
                         תקין? → כניסה
                         שגוי → ניסיון נוסף
                         3 כשלונות → ניתוק + חסימה זמנית (30 דקות)
```

### כללי אבטחה

| נושא | הגדרה |
|------|--------|
| **Caller ID** | זיהוי ראשוני — חיפוש ב-`users.phone` |
| **PIN** | 4 ספרות, מוצפן ב-DB, מוגדר על ידי המשתמש **באתר** |
| **כניסה מהירה** | המשתמש יכול לבחור באתר "אפשר כניסה ללא PIN מהמכשיר שלי" (`allow_ivr_no_pass = true`) |
| **מספר חסוי** | "הקש מספר נייד + PIN" |
| **שכחת PIN** | "לאיפוס PIN היכנס להגדרות החשבון באתר הפנקס." — אין איפוס דרך הטלפון. |
| **3 כשלונות PIN** | ניתוק + חסימה ל-30 דקות. שמירת `ivr_failed_attempts` + `ivr_blocked_until` ב-DB. |

## 4. מנגנון האודיו

### א. Cache מקומי (עלות כמעט אפסית)

1. לפני כל שליחה ל-Google TTS, השרת בודק אם קובץ MP3 קיים ב-Cache.
2. **קיים** → שולח ישירות. **לא קיים** → יוצר + שומר + שולח.
3. **שם הקובץ:** מבוסס על `hash(טקסט) + user_id + updated_at`. כאשר פרופיל מתעדכן — updated_at משתנה → קובץ ישן לא נמצא → נוצר חדש.
4. **מילים נפוצות** (ערים, ישיבות, ספירות) — נוצרות פעם אחת בלבד לכל תולדות המערכת.

### ב. הרכבה אטומית (Atomic Audio)
משפטים ארוכים מורכבים מקטעים קבועים + קטעים דינמיים, לדוגמה:
- `welcome.mp3` + `[שם].mp3` + `you_have.mp3` + `2.mp3` + `new_offers.mp3`

### ג. מילון הגייה — `phonetic-map.json` (קובץ נפרד)

| מה קיים | מה נשלח ל-TTS |
|---------|---------------|
| `pending` | ממתין לתשובה |
| `active` | פעיל |
| `waiting_for_shadchan` | בטיפול שדכנית |
| `rejected` | הצעה שנסגרה |
| `ת"ת` | תלמוד תורה |
| `ל"ת` | ליטאי |
| `חב"ד` | חַבָּד |
| `פוניבז'` | פוני-בז |
| *(להרחבה)* | *(קובץ JSON — לא קוד)* |

### ד. פיסוק פרוזודי
לפני שליחת טקסט חופשי ל-TTS, השרת משתיל פסיקים ונקודות במקומות אסטרטגיים כדי לאלץ השהיות נשימה טבעיות.

### ה. Fallback (TTS לא זמין)
אם ה-API של Google לא מגיב תוך 3 שניות — המערכת משמיעה קובץ מוקלט מראש:  
`"הפעולה נקלטה בהצלחה"` / `"אירעה תקלה, אנא נסה שוב מאוחר יותר."`

## 5. זרימת השיחה — תפריט מלא

### כניסה ומבזק סטטוס
מיד לאחר אימות, המשתמש שומע:

> "שלום [שם]. יש לך **[X] הצעות חדשות**, **[Y] בקשות קשר ממתינות לתשובתך**, ו-**[Z] בקשות תמונה**."

*(הודעות מערכת לא נכללות במבזק — ראה סעיף 6.5)*

### מפת התפריט

```
תפריט ראשי
├── 1  הצעות חדשות ממנוע ההתאמות
├── 2  בקשות קשר שנשלחו אליי (ממתינות לתשובתי)
├── 3  סטטוס פניות שיצאו ממני
├── 4  ניהול תמונות
├── 5  הודעות חשובות
├── 9  הגדרות (שינוי PIN)
├── 0  תמיכה
├── *  חזרה לתפריט קודם
└── #  חזרה לתפריט הראשי (מכל שלב)
```

## 6. פירוט כל מקש ותת-תפריט

### 6.1 — מקש `1`: הצעות חדשות ממנוע ההתאמות

**מה זה:** פרופילים שמנוע ההתאמות מציע לי לבחון — עם אנשים שעדיין לא יצרתי איתם כל קשר.

**זרימה:**

```
"יש [X] הצעות חדשות. מתחילים עם ההצעה הראשונה."
    │
    ▼
[הקראת כרטיסייה — שכבות]
  שכבה חובה:    שם/קוד, גיל, עיר, מוסד לימודים/ישיבה/סמינר
  4 = שכבת רשות: מוצא, סגנון משפחתי, מקצוע
  5 = תיאור חופשי של השדכנית
    │
    ▼
"הקש 1 — מעוניין.  2 — לא מעוניין.  8 — דלג.  # — תפריט ראשי."
    │
    ├── 1 → שליחת פנייה [API: POST /connect] → "פנייתך נקלטה. נעבור להצעה הבאה." → המשך לולאה
    ├── 2 → דחיית ההצעה (הסתרה) [API: POST /api/hide-profile] → "הצעה הוסרה." → הבאה
    └── 8 → דלג → הצעה הבאה
```

**סיום לולאה:** "סיימנו את כל ההצעות החדשות." → חזרה לתפריט ראשי.

---

### 6.2 — מקש `2`: בקשות קשר שנשלחו אליי

**מה זה:** אנשים ספציפיים שכבר שלחו לי פנייה ומחכים לתשובתי. מקביל ל-`GET /my-requests` (status=`pending`, receiver=אני).

> **חשוב:** זה שונה מ"הצעות". כאן מישהו יזם וממתין — לא מגיב = הוא תקוע.

**זרימה:**

```
"יש [Y] בקשות קשר ממתינות לתשובתך."
    │
    ▼
"בקשה מספר 1: [שם/קוד], גיל [X], עיר [Y]."
"הקש 4 לשמיעת פרטים נוספים."
"הקש 1 — מסכים.  2 — לא מסכים.  8 — טפל מאוחר יותר."
    │
    ├── 1 → אישור [API: POST /approve-request] → "הבקשה אושרה." → הבאה
    ├── 2 → דחייה [API: POST /reject-request] → "הבקשה נדחתה." → הבאה
    └── 8 → דלג → הבאה (הבקשה נשארת פתוחה)
```

---

### 6.3 — מקש `3`: סטטוס פניות שיצאו ממני

**מה זה:** הצגת סטטוס עדכני של פניות שאני שלחתי — לפי מחזור חיי ה-Connection.

**זרימה לפי סטטוס:**

**`pending` — שלחתי, ממתין לתשובה:**
> "ביום [יום] פנית ל[שם]. הצד השני טרם הגיב. הקש 1 לביטול הפנייה. הקש 8 להמשך."
- `1` → `POST /cancel-request` → "הפנייה בוטלה."

**`active` — שניהם הביעו עניין ראשוני:**

*מצב א' — טרם נתתי אישור סופי:*
> "ההצעה עם [שם] פעילה — שניכם הביעו עניין. לאישור סופי ומעבר לשדכנית הקש 1. לביטול הפנייה הקש 2."
- `1` → `POST /finalize-connection`
- `2` → `POST /cancel-active-connection` → "הפנייה בוטלה."

*מצב ב' — כבר נתתי אישור סופי, מחכה לצד השני:*
> "ההצעה עם [שם] פעילה — כבר אישרת להמשיך. ממתין לאישור הצד השני. הקש 8 להמשך."

**`waiting_for_shadchan` — בטיפול:**
> "התיק עם [שם] הועבר לטיפול השדכנית. אין צורך בפעולה נוספת כרגע. הקש 8 להמשך."

---

### 6.4 — מקש `4`: ניהול תמונות

**מה זה:** ניהול בקשות לחשיפת תמונות — אישור/דחייה/שליחה. מקביל ל-`photo_approvals`.

**זרימה:**

```
"יש [Z] בקשות תמונה ממתינות."
    │
    ▼
"[שם/קוד] מבקשת לראות את תמונתך."
"הקש 1 — הסכם לחשיפה.  2 — דחה את הבקשה."
    │
    ├── 1 → [API: POST /respond-photo-request {status:'approved'}] → "הסכמה נשלחה."
    └── 2 → [API: POST /respond-photo-request {status:'rejected'}] → "הבקשה נדחתה."
```

**שליחת בקשה:**
> "לשליחת בקשה לצפייה בתמונת הצד שאיתו יש לך חיבור פעיל, הקש 3."
- `3` → `POST /request-photo-access`

---

### 6.5 — מקש `5`: הודעות חשובות (מסוננות)

**עיקרון מנחה:** **קלילות וזולות.** לא כל הודעה שווה שמיעה בטלפון.

**סינון — מה כן נכלל:**

| סוג הודעה | כן/לא | נימוק |
|-----------|-------|--------|
| "הצד השני ביטל פנייה אליך" | ✅ | פעיל — רלוונטי לניהול השידוך |
| "הצד השני ביטל חיבור פעיל" | ✅ | פעיל — רלוונטי |
| "השדכנית שלחה לך הודעה אישית" | ✅ | בעל ערך גבוה |
| "הפרופיל שלך אושר" | ❌ | לא דורש פעולה — לא קולי |
| "ברוכים הבאים / הרשמה הצליחה" | ❌ | לא רלוונטי לטלפון |
| "תזכורת: מלא פרופיל" | ❌ | לא ניתן לביצוע בטלפון |

**זרימה:**
> "יש הודעה חדשה: [תוכן ההודעה המסוננת]."  
> "להאזנה חוזרת הקש 1. להודעה הבאה הקש 8."

אם אין הודעות מסוננות: "אין הודעות חדשות הדורשות תשומת לבך." → חזרה לתפריט ראשי.

---

### 6.6 — מקש `9`: הגדרות (שינוי PIN)

**זרימה:**
> "להחלפת קוד הכניסה הנוכחי, הקש את הקוד הנוכחי."  
> ✅ תקין → "הקש קוד חדש בן 4 ספרות." → "הקש שוב לאישור." → "הקוד עודכן בהצלחה."  
> ❌ שגוי → "קוד שגוי. נסה שוב." (עד 3 פעמים)

---

### 6.7 — מקש `0`: תמיכה

**זרימה:**
> "השאר הודעה קצרה לצוות התמיכה, ואנו נחזור אליך. דבר לאחר הצפצוף."  
→ הקלטה קולית → שמירה בשרת → שליחת התראה לצוות.

## 7. ניהול חוויית משתמש ושגיאות

### זיכרון שיחה (Session Management)

- שיחה שהתנתקה באמצע הקראת פרופיל — בחיוג הבא:  
  > "זיהינו שהשיחה הקודמת הופסקה. להמשך מאותה נקודה הקש 1. לתפריט הראשי הקש 2."
- שמירה ב-`ivr_sessions`: `user_id`, `last_menu`, `last_item_id`, `updated_at`.

### Timeouts

| מצב | התנהגות |
|-----|---------|
| אין הקשה במשך 8 שניות | חזרה על אפשרויות התפריט |
| 3 חזרות ללא תגובה | "לא זוהתה הקשה, נתראה בקרוב." → ניתוק |
| 5 דקות ללא הקשה | ניתוק יזום (מניעת חיוב על קו פתוח) |

**הבהרה:** 5 הדקות הן ללא הקשה — לא סה"כ זמן שיחה. שיחה שבה המשתמש מאזין להצעות רצופות יכולה להיות ארוכה ללא הגבלה.

### הקשה שגויה

> "מקש לא מוכר." → חזרה לתפריט הקודם.  
אחרי 3 הקשות שגויות ברציפות → חזרה לתפריט הקודם.

### קיצורי דרך גלובליים

- `*` — חזרה שלב אחורה בכל נקודה
- `#` — חזרה לתפריט הראשי בכל נקודה
- `5` בכל הקראת פרופיל — דלג ישר לתפריט הפעולות

## 8. אינטגרציה עם ה-API הקיים

ה-IVR **לא** מכיל לוגיקה עסקית. כל פעולה היא Webhook לאחד מה-endpoints הקיימים:

| פעולה ב-IVR | API endpoint |
|------------|--------------|
| שליחת פנייה | `POST /connect` |
| ביטול פנייה pending | `POST /cancel-request` |
| אישור בקשת קשר נכנסת | `POST /approve-request` |
| דחיית בקשת קשר נכנסת | `POST /reject-request` |
| אישור סופי | `POST /finalize-connection` |
| ביטול חיבור active | `POST /cancel-active-connection` |
| אישור/דחיית תמונה | `POST /respond-photo-request` |
| שליחת בקשת תמונה | `POST /request-photo-access` |
| הסתרת הצעה | `POST /api/hide-profile` |
| שליפת הצעות | `GET /matches` |
| שליפת בקשות נכנסות | `GET /my-requests` |
| שליפת סטטוס פניות | `GET /my-connections` + `GET /my-sent-requests` |
| שליפת בקשות תמונה | `GET /pending-photo-requests` |
| שליפת הודעות | `GET /my-messages` (מסוננות) |

**אימות ה-Webhooks:** קריאות ה-IVR לשרת ייבוצעו עם `ivr_service_token` ייעודי בכותרת ה-Authorization.

**לוגים:** כל פעולה תירשם ב-`ivr_logs` ותשויך ל-`activity_log` הקיים.

## 9. עדכוני מסד נתונים

### 9.1 — שינויים בטבלת `users`

| שדה | סוג | ברירת מחדל | תפקיד |
|-----|-----|-----------|--------|
| `ivr_pin` | varchar(255) | NULL | קוד PIN מוצפן (bcrypt) |
